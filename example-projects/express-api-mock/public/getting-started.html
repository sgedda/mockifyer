🚀 Getting Started

Get up and running with Mockifyer in minutes. Learn the simplest way to record and replay API calls.

Quick Start

Mockifyer works by intercepting HTTP requests made by your application. In its simplest form, you just need to:

  1. Install the package
  2. Initialize Mockifyer before making any HTTP calls
  3. Use your HTTP client as normal - Mockifyer handles the rest!

Platform Guides

Node Service

Perfect for Node.js services, Express APIs, and backend applications. Mockifyer integrates seamlessly with Axios, Fetch, and other HTTP clients.

1Install Mockifyer

Install the packages you need based on your HTTP client:

# For Axios npm install @sgedda/mockifyer-core @sgedda/mockifyer-axios # For Fetch npm install @sgedda/mockifyer-core @sgedda/mockifyer-fetch # For both Axios and Fetch npm install @sgedda/mockifyer-core @sgedda/mockifyer-axios @sgedda/mockifyer-fetch

2Initialize Mockifyer

Option A - Direct Usage with Axios: Use the HTTP client returned by setupMockifyer directly:

import { setupMockifyer } from '@sgedda/mockifyer-axios'; import path from 'path'; // Setup Mockifyer - returns an HTTP client you can use directly const axios = setupMockifyer({ mockDataPath: path.join(__dirname, 'mock-data'), recordMode: process.env.MOCKIFYER_RECORD === 'true' }); // Use it like normal axios const response = await axios.get('https://api.example.com/users'); // Your API calls will be automatically intercepted!

Option B - Global Axios Patching: Patch global axios to intercept all axios calls:

import { setupMockifyer } from '@sgedda/mockifyer-axios'; // Initialize before importing axios setupMockifyer({ mockDataPath: './mock-data', recordMode: process.env.MOCKIFYER_RECORD === 'true', useGlobalAxios: true // Patch global axios }); // Now import axios - it's automatically patched import axios from 'axios'; // Use axios normally - requests are intercepted const response = await axios.get('https://api.example.com/users');

Option C - Fetch: For fetch-based applications:

import { setupMockifyer } from '@sgedda/mockifyer-fetch'; // Initialize before using fetch setupMockifyer({ mockDataPath: './mock-data', recordMode: process.env.MOCKIFYER_RECORD === 'true', useGlobalFetch: true // Patch global fetch }); // Use fetch() normally - requests are intercepted const response = await fetch('https://api.example.com/users');

3Record API Responses (Optional)

To capture real API responses for later use, enable recording mode:

import { setupMockifyer } from '@sgedda/mockifyer-axios'; // Set environment variable or pass recordMode: true process.env.MOCKIFYER_RECORD = 'true'; // Or in setupMockifyer config: setupMockifyer({ mockDataPath: './mock-data', recordMode: true // Enable recording }); // Make your API calls - responses will be saved automatically const response = await axios.get('https://api.example.com/users'); // Response is saved to: mock-data/2025-11-23_10-53-52_GET_api_example_com_users.json

4Use Mocked Responses

Once you've recorded responses, disable recording mode to use the saved mocks:

import { setupMockifyer } from '@sgedda/mockifyer-axios'; setupMockifyer({ mockDataPath: './mock-data', recordMode: false // Use saved mocks instead of recording }); // This call will use the saved mock instead of making a real API call const response = await axios.get('https://api.example.com/users'); // ✅ Instant response from mock file, no network call!

✨ What to Expect

  • Mock Files Created: When recording, JSON files are automatically created in your mock-data directory
  • Descriptive Filenames: Files are named with timestamp, method, and endpoint (e.g., 2025-11-23_10-53-52_GET_api_example_com_users.json)
  • Searchable Data: All mock files are JSON files in your project, fully searchable in your IDE
  • Complete Request/Response: Each file contains the full request (URL, method, headers, params) and response (status, data, headers)
  • Instant Responses: When using mocks, responses are instant with no network latency

Complete Example

Here's a complete example showing both recording and replay with Axios:

import { setupMockifyer } from '@sgedda/mockifyer-axios'; import path from 'path'; async function main() { // Step 1: Record API responses console.log('Recording API calls...'); const recordAxios = setupMockifyer({ recordMode: true, mockDataPath: path.join(__dirname, 'mock-data') }); // Make a real API call - response will be saved automatically const response1 = await recordAxios.get('https://jsonplaceholder.typicode.com/posts/1'); console.log('Recorded:', response1.data); // File saved: mock-data/2025-11-23_10-53-52_GET_jsonplaceholder_typicode_com_posts_1.json // Step 2: Use recorded mocks console.log('Using recorded mocks...'); const mockAxios = setupMockifyer({ recordMode: false, mockDataPath: path.join(__dirname, 'mock-data') }); // This uses the saved mock instead of making a real API call const response2 = await mockAxios.get('https://jsonplaceholder.typicode.com/posts/1'); console.log('From mock:', response2.data); // Same data, instant response! } main();

Environment Variables

You can also configure Mockifyer using environment variables:

# Enable Mockifyer MOCKIFYER_ENABLED=true # Path to mock data directory MOCKIFYER_PATH=./mock-data # Enable recording mode MOCKIFYER_RECORD=true

Then initialize with minimal config (for Axios):

import { setupMockifyer } from '@sgedda/mockifyer-axios'; // Reads configuration from environment variables and patches global axios setupMockifyer({ mockDataPath: process.env.MOCKIFYER_PATH || './mock-data', recordMode: process.env.MOCKIFYER_RECORD === 'true', useGlobalAxios: true });

React Native / Expo

Mockifyer works seamlessly with React Native and Expo applications. It supports both development (with recording) and production (with bundled mocks) modes.

1Install Dependencies

npm install @sgedda/mockifyer-core @sgedda/mockifyer-fetch expo-file-system

@sgedda/mockifyer-core is required as a dependency. expo-file-system is required for the Expo FileSystem provider used in React Native.

2Configure Metro Bundler

Why is this needed? Mockifyer uses Node.js built-in modules like fs and path for filesystem operations. These modules don't exist in React Native, so Metro bundler will fail when trying to bundle Mockifyer code that imports them.

The Metro configuration stubs these modules with empty placeholders, allowing Metro to bundle successfully. At runtime, Mockifyer detects that these modules aren't available and gracefully falls back to React Native-compatible providers (like Expo FileSystem).

Without this configuration, you'll see errors like:

Error: Unable to resolve module "fs" from "packages/mockifyer-core/src/providers/filesystem-provider.ts" Error: Unable to resolve module "path" from "packages/mockifyer-fetch/src/index.ts"

Solution: Use the provided helper function to automatically configure Metro:

// metro.config.js const { getDefaultConfig } = require('expo/metro-config'); const { configureMetroForMockifyer } = require('@sgedda/mockifyer-fetch/metro-config'); const config = getDefaultConfig(__dirname); module.exports = configureMetroForMockifyer(config);

This helper automatically stubs fs, path, assert, and util modules that Mockifyer and its dependencies use. It preserves your existing Metro configuration and has zero impact on your bundle size (Metro config runs at build time, not runtime).

3Initialize Mockifyer

Use the React Native helper function for automatic provider selection:

// mockifyer-setup.ts import { setupMockifyerForReactNative } from '@sgedda/mockifyer-fetch/react-native'; export async function initializeMockifyer() { return await setupMockifyerForReactNative({ isDev: __DEV__, // Pass React Native's __DEV__ variable mockDataPath: 'mock-data', bundledDataPath: './assets/mock-data', // For production builds recordMode: process.env.MOCKIFYER_RECORD === 'true', }); }

Then initialize in your app:

// App.tsx import { useEffect } from 'react'; import { initializeMockifyer } from './mockifyer-setup'; export default function App() { useEffect(() => { initializeMockifyer(); }, []); // Your app code... }

4Development Mode (Recording)

In development, Mockifyer uses the Expo FileSystem provider and can record API responses:

// Start with recording enabled MOCKIFYER_RECORD=true npm start // Or in your setup: setupMockifyerForReactNative({ isDev: true, mockDataPath: 'mock-data', recordMode: true, // Enable recording }); // Make API calls - responses are saved automatically const response = await fetch('https://api.example.com/users'); // Saved to device filesystem: FileSystem.documentDirectory + 'mock-data/'

Hybrid Provider: In development, you can use the Hybrid Provider to save mocks to both device filesystem AND your project folder simultaneously. See the example project for details.

5Production Mode (Bundled Mocks)

For production builds, generate a TypeScript bundle from your recorded mocks:

# Generate bundle from recorded mocks npm run generate:build-data # Or use CLI directly: npx @sgedda/mockifyer-core generate-bundle \\ --input ./mock-data \\ --output ./assets/mock-data.ts \\ --format typescript

The helper automatically uses the Memory provider with bundled data in production:

// In production (isDev: false), automatically uses: // - Memory provider // - Bundled TypeScript file (./assets/mock-data.ts) // - No filesystem access needed

✨ What to Expect

  • Development: Uses Expo FileSystem provider - mocks saved to device filesystem
  • Production: Uses Memory provider with bundled TypeScript file - no filesystem needed
  • Automatic Switching: Provider selection is automatic based on isDev flag
  • Metro Config: Helper function handles all Metro configuration automatically
  • Zero Bundle Impact: Metro config runs at build time, not included in bundle

Complete Example

Here's a complete React Native setup:

// metro.config.js const { getDefaultConfig } = require('expo/metro-config'); const { configureMetroForMockifyer } = require('@sgedda/mockifyer-fetch/metro-config'); module.exports = configureMetroForMockifyer(getDefaultConfig(__dirname)); // mockifyer-setup.ts import { setupMockifyerForReactNative } from '@sgedda/mockifyer-fetch/react-native'; export async function initializeMockifyer() { return await setupMockifyerForReactNative({ isDev: __DEV__, mockDataPath: 'mock-data', bundledDataPath: './assets/mock-data', recordMode: __DEV__ && process.env.MOCKIFYER_RECORD === 'true', }); } // App.tsx import { useEffect } from 'react'; import { initializeMockifyer } from './mockifyer-setup'; export default function App() { useEffect(() => { initializeMockifyer().then(() => { console.log('Mockifyer initialized'); }); }, []); const fetchData = async () => { const response = await fetch('https://api.example.com/users'); const data = await response.json(); console.log('Data:', data); }; return ( // Your app UI... ); }

Using the Dashboard

The Mockifyer Dashboard is a web interface for viewing, editing, and managing your mock data files. It provides a visual way to explore all your recorded API responses.

1Start the Dashboard

Run the dashboard using npx (no installation needed):

# Auto-detect mock data path npx @sgedda/mockifyer-dashboard # Or specify the path explicitly npx @sgedda/mockifyer-dashboard --path ./mock-data # Custom port (default is 3001) npx @sgedda/mockifyer-dashboard --port 8080

The dashboard will automatically open in your browser at http://localhost:3001.

2Browse Your Mocks

The dashboard displays all your mock files with:

  • Endpoint URL - The API endpoint that was called
  • HTTP Method - GET, POST, PUT, DELETE, etc.
  • File Size - Size of the mock file
  • Last Modified - When the mock was recorded
  • Search & Filter - Find specific mocks quickly

3Edit Mock Data

Click any mock file to view and edit it:

  • View Request Details - See the full request (URL, headers, query params, body)
  • Edit Response Data - Modify the response using the built-in JSON editor
  • Save Changes - Save edited mocks back to the file
  • Delete Mocks - Remove unwanted mock files

Changes are saved directly to your mock files, so they're immediately available to your application.

4View Statistics

The dashboard provides insights into your mock data:

  • Total Files - Number of recorded mocks
  • Total Size - Combined size of all mock files
  • Most Used Endpoints - See which APIs you call most frequently
  • HTTP Methods Distribution - Breakdown by GET, POST, etc.
  • Status Codes Distribution - See response status code patterns

React Native Integration

For React Native projects, the dashboard works seamlessly with Metro's sync middleware:

# Terminal 1: Start Metro bundler npm start # Terminal 2: Start Dashboard npx @sgedda/mockifyer-dashboard --path ./mock-data

When using the Hybrid Provider in development, mocks saved by your app automatically sync to your project folder, and the dashboard picks them up instantly. No manual sync needed!

Dashboard Options

# Available options npx @sgedda/mockifyer-dashboard \\ --path ./mock-data # Path to mock data directory --port 8080 # Custom port (default: 3001) --host 0.0.0.0 # Bind to all interfaces (for remote access) --no-open # Don't open browser automatically

✨ Dashboard Features

  • Real-time Updates - See new mocks as they're recorded
  • JSON Editor - Edit mock responses with syntax highlighting
  • Search & Filter - Quickly find specific endpoints or methods
  • Request Flow Visualization - See how requests relate to each other
  • Statistics Dashboard - Understand your API usage patterns

Deterministic Dates with Sinon Fake Timers

Mockifyer provides getCurrentDate() for getting manipulated dates, but for deterministic dates where new Date() and Date.now() return fake time globally, you can use Sinon fake timers alongside Mockifyer in Node.js environments.

Why Use Fake Timers?

Mockifyer's getCurrentDate() returns manipulated dates, but it doesn't affect global Date() or Date.now(). If your code uses new Date() or Date.now() directly, you'll need fake timers to make those return deterministic dates.

Use fake timers when:

  • Your code calls new Date() or Date.now() directly
  • You need setTimeout/setInterval to work with fake time
  • You want deterministic dates across your entire application
  • You're testing time-dependent logic

Setting Up Sinon Fake Timers

Here's how to set up Sinon fake timers alongside Mockifyer in your service:

import sinon from 'sinon'; import { useFakeTimers } from 'sinon'; import { setupMockifyer } from '@sgedda/mockifyer-fetch'; import { getCurrentDate } from '@sgedda/mockifyer-core'; // Initialize Mockifyer with date manipulation const mockifyer = setupMockifyer({ mockDataPath: './mock-data', recordMode: false, dateManipulation: { fixedDate: '2024-12-25T00:00:00.000Z', }, }); // Setup Sinon fake timers for global Date manipulation const fixedDate = '2024-12-25T00:00:00.000Z'; const clock = useFakeTimers({ now: new Date(fixedDate), // Same date as Mockifyer toFake: ['Date', 'setTimeout', 'setInterval'], }); // Now both new Date() and getCurrentDate() return the same fake date const globalDate = new Date(); // Returns 2024-12-25 (from Sinon) const mockifyerDate = getCurrentDate(); // Returns 2024-12-25 (from Mockifyer) // Your service code can use deterministic dates function checkSubscriptionStatus(subscription: any) { const currentDate = new Date(); // Uses fake time if (subscription.expiresAt < currentDate) { return 'expired'; } return 'active'; } // When done, restore real timers // clock.restore();

When to Use What

Mockifyer's getCurrentDate():

  • ✅ Returns manipulated dates
  • ✅ Works everywhere (Node.js, React Native, Browser)
  • ❌ Doesn't affect global Date() or Date.now()

Sinon/Jest fake timers:

  • ✅ Makes new Date() and Date.now() return fake time globally
  • ✅ Controls setTimeout/setInterval
  • ✅ Allows time progression (clock.tick() or jest.advanceTimersByTime())
  • ✅ Perfect for deterministic dates across your entire application

Environment compatibility:

  • Node.js: Use Sinon fake timers
  • React Native: Use Mockifyer's getCurrentDate() instead
  • Browser: Use Mockifyer's getCurrentDate() instead

✨ Best Practices

  • Use Mockifyer's getCurrentDate(): When you can - it's simpler and works everywhere
  • Use Sinon fake timers: When your code uses new Date() or Date.now() directly in Node.js
  • Keep dates in sync: Use the same date for Mockifyer config and fake timers
  • Clean up: Always restore fake timers when done (clock.restore())
  • For React Native/Browser: Use Mockifyer's getCurrentDate() instead

🧪 Automatic Test Generation

Mockifyer can automatically generate unit tests when recording mocks. This feature creates test files that use your recorded mock data, ensuring tests are always in sync with your mocks.

Features

  • ✅ Automatic test generation - Tests created when mocks are saved
  • ✅ Multiple frameworks - Supports Jest, Vitest, and Mocha
  • ✅ Realistic data - Uses recorded real API responses
  • ✅ Zero setup - Generated tests include all setup code
  • ✅ Smart grouping - Organize tests by endpoint, scenario, or file
  • ✅ GraphQL support - Automatically detects and handles GraphQL requests

Enable Test Generation

Enable test generation in your Mockifyer configuration:

import { setupMockifyer } from '@sgedda/mockifyer-axios'; // or @sgedda/mockifyer-fetch setupMockifyer({ mockDataPath: './mock-data', recordMode: true, useGlobalAxios: true, // or useGlobalFetch: true // Enable test generation generateTests: { enabled: true, framework: 'jest', // 'jest' | 'vitest' | 'mocha' outputPath: './tests/generated', testPattern: '{endpoint}.test.ts', includeSetup: true, groupBy: 'endpoint' // 'endpoint' | 'scenario' | 'file' } });

How It Works

1Record Mocks

Make API calls in your application while recording mode is enabled:

// Your app code const response = await fetch('https://api.example.com/users/1'); const user = await response.json();

When Mockifyer saves the mock, it automatically generates a test file.

2Generated Test

Mockifyer creates a test file with realistic assertions:

// tests/generated/api/users/1.test.ts - AUTO-GENERATED import { setupMockifyer } from '@sgedda/mockifyer-fetch'; describe('/api/users/1', () => { let mockifyer: any; beforeAll(() => { mockifyer = setupMockifyer({ mockDataPath: './mock-data', recordMode: false, useGlobalFetch: true, scenario: 'default' }); }); it('should GET /api/users/1', async () => { const response = await fetch('https://api.example.com/users/1'); const data = await response.json(); expect(response.status).toBe(200); expect(data).toEqual({ id: 1, name: 'John Doe', email: 'john@example.com' }); }); });

3Run Tests

Run your generated tests:

npm test # ✅ Tests pass - uses mock, no real API call!

Configuration Options

generateTests.enabled

Enable or disable test generation.

generateTests: { enabled: true // Default: false }

generateTests.framework

Test framework to use.

generateTests: { framework: 'jest' // 'jest' | 'vitest' | 'mocha' }

generateTests.outputPath

Where to save generated tests.

generateTests: { outputPath: './tests/generated' // Default: './tests/generated' }

generateTests.groupBy

How to organize tests.

  • 'endpoint' → tests/generated/api/users/1.test.ts
  • 'scenario' → tests/generated/scenarios/happy-path/api/users/1.test.ts
  • 'file' → tests/generated/api_users_1.test.ts

GraphQL Support

GraphQL requests are automatically detected and handled:

// Generated test for GraphQL describe('/graphql', () => { it('should execute GetUsers query', async () => { const response = await fetch('https://api.example.com/graphql', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ query: `query GetUsers($limit: Int) { users(limit: $limit) { id name } }`, variables: { limit: 10 } }) }); const data = await response.json(); expect(response.status).toBe(200); expect(data.data.users).toBeDefined(); }); });

Using Test Helper for Zero Setup

Combine with @sgedda/mockifyer-test-helper for complete automation:

// Install test helper npm install @sgedda/mockifyer-test-helper
// jest.setup.ts import { jestSetup } from '@sgedda/mockifyer-test-helper'; jestSetup(); // Auto-setup for all tests - zero configuration!
// Your tests - zero setup! import { render, fireEvent, waitFor } from '@testing-library/react-native'; import App from '../App'; describe('App Component', () => { it('should display posts', async () => { const { getByText } = render(<App />); fireEvent.press(getByText('Load Posts')); // Uses REALISTIC data from recorded API calls! await waitFor(() => { expect(getByText('sunt aut facere repellat')).toBeTruthy(); }); }); });

Scenario Switching in Tests

Use different scenarios in your tests:

import { useScenario } from '@sgedda/mockifyer-test-helper'; describe('Different scenarios', () => { it('should handle empty state', () => { useScenario('empty-posts'); const { getByText } = render(<App />); fireEvent.press(getByText('Load Posts')); expect(getByText('No posts available')).toBeTruthy(); }); it('should handle errors', () => { useScenario('error-500'); const { getByText } = render(<App />); fireEvent.press(getByText('Load Posts')); expect(getByText('Something went wrong')).toBeTruthy(); }); });
